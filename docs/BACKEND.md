# Backend architecture

The backend is a v1 CRUD API in front of Postgres, written in Go, that
serves the same `GraphNode`/`GraphEdge` shapes the frontend already defines
in `src/data/types.ts`. It also serves the built frontend as static files,
so in production it's the only process that runs.

Language/stack choice: Go + [sqlc](https://sqlc.dev/) + Postgres
(`pgx/v5`). The data model is two flat tables with a handful of string
enums and one foreign-key relationship — not complex enough for
type-sharing with the frontend to matter, so the deciding factors were
container footprint and toolchain simplicity: a static Go binary keeps the
final Docker image small, and sqlc means writing plain SQL instead of
fighting an ORM for what is, for now, straightforward CRUD.

## Directory layout (`backend/`)

```
backend/
  cmd/server/main.go       # entrypoint: wires config, db, router, static file serving
  internal/
    config/config.go       # env var loading
    db/                     # sqlc input (schema.sql, queries.sql) + generated code
    api/                    # HTTP handlers (net/http, no framework)
  sqlc.yaml                 # sqlc codegen config
  go.mod / go.sum
```

There's no `internal/store` or repository-interface layer — handlers call
the sqlc-generated `*db.Queries` directly. With one Postgres engine and no
alternate backing store, that indirection wouldn't be pulling its weight;
add it if a second implementation (e.g. an in-memory fake for tests) ever
becomes real.

## Data layer (`internal/db/`)

This is a mix of hand-written and generated files:

- **`schema.sql`** — hand-written. Creates the `mindmap` Postgres schema
  (not `public`) and the `nodes`/`edges` tables inside it. Every statement
  is `CREATE ... IF NOT EXISTS`, and `migrate.go` (`Migrate()`) runs this
  file on every server startup via `pgxpool.Pool.Exec` — there's no
  migration framework or version table. This is enough for a two-table v1;
  if schema changes stop being purely additive (renames, column drops,
  backfills), replace this with a real migration tool rather than trying to
  make `IF NOT EXISTS` do more than it can.
- **`queries.sql`** — hand-written. One `-- name: X :one|:many|:exec` block
  per query, using `sqlc.arg(name)`/`sqlc.narg(name)` for named/nullable
  parameters. All table references are schema-qualified (`mindmap.nodes`,
  not `nodes`) so queries are correct regardless of the connection's
  `search_path`.
- **`models.go`, `querier.go`, `queries.sql.go`, `db.go`** — generated by
  `sqlc generate`; never hand-edit these (they say so at the top, and your
  changes will be silently overwritten on the next generate).
- **`connect.go`, `migrate.go`** — hand-written glue: `Connect()` opens the
  pgx pool, pings it, and runs `Migrate()`; both live in package `db`
  alongside the generated code since `Migrate()` needs `//go:embed
  schema.sql`, which only works on a file in the same directory as the
  embedding source.

### Adding or changing a query

1. Edit `schema.sql` and/or `queries.sql` in `internal/db/`.
2. Regenerate:
   ```bash
   cd backend
   export PATH="$PATH:$(go env GOPATH)/bin"   # if sqlc isn't already on PATH
   sqlc generate
   ```
   (Install once with `go install github.com/sqlc-dev/sqlc/cmd/sqlc@latest`
   if you don't have it.)
3. `go build ./...` to catch anything the new/changed generated signatures
   break in `internal/api/`.

**Gotcha:** because the tables live in the `mindmap` schema rather than
`public`, sqlc prefixes the generated struct names with the schema to avoid
collisions — the model types are `db.MindmapNode` / `db.MindmapEdge`, not
`db.Node` / `db.Edge`. Query-specific types (`CreateNodeParams`,
`UpdateEdgeParams`, etc.) aren't affected, since those are named after the
query, not the table.

## API layer (`internal/api/`)

Plain `net/http` using Go 1.22+'s method+pattern routing
(`mux.HandleFunc("GET /api/nodes/{id}", ...)`) — no router dependency.
`router.go` builds the `*http.ServeMux` and holds the `Server` struct
(currently just wraps `*db.Queries`); `nodes.go`/`edges.go` hold the
handlers, one file per resource.

Each resource follows the same five-handler shape — use `nodes.go` as the
template for adding a new resource:

- `list*` — calls `List*(ctx)`. **Must** convert a `nil` result to an empty
  slice before encoding (`db.Queries.List*` returns a nil Go slice when
  there are no rows, which encodes as JSON `null`; the frontend calls
  `.find()`/`.filter()` on the response and will throw on `null`). Both
  `listNodes` and `listEdges` do this — copy the pattern.
- `get*` — calls `Get*(ctx, id)`; maps `pgx.ErrNoRows` to 404 via
  `errors.Is`.
- `create*` — decodes a request body, validates required fields, calls
  `Create*Params`-shaped insert, returns 201.
- `update*` — same shape as create but takes the id from `r.PathValue("id")`
  (not the body — the body's `id` field, if present, is ignored), maps
  `pgx.ErrNoRows` to 404.
- `delete*` — calls `Delete*(ctx, id)`, returns 204. No existence check
  before deleting (Postgres just deletes zero rows if the id doesn't
  exist, which is a fine no-op for `DELETE`).

`json.go` holds the three shared helpers (`writeJSON`, `writeError`,
`decodeJSON`) every handler uses — errors always come back as
`{"error": "..."}`.

`spa.go` (`SPAHandler`) serves the built frontend and isn't really part of
the CRUD API — see "Serving the frontend" below.

### Request/response shapes vs. the frontend contract

The JSON field names in `nodeRequest`/`edgeRequest` (in `nodes.go`/`edges.go`)
and the sqlc-generated model structs are deliberately kept identical to
`GraphNode`/`GraphEdge` in `src/data/types.ts` — same field names, same
optionality (nullable Postgres columns map to Go `*string`, which
marshals as JSON `null`/absent-or-present the way the frontend's optional
fields expect). If you add a field to a `GraphNode`/`GraphEdge` in the
frontend, add the matching column + query changes here and keep the JSON
tags matching; there's no shared type generation between the two, so
this has to be kept in sync by hand.

Note that `edges` has a server-generated-looking but actually
**client-supplied** `id` (a string slug, same as node ids) — the frontend
must supply one on create. This was added purely so individual edges are
addressable for update/delete; it doesn't exist elsewhere in the graph
model.

## Config (`internal/config/`)

Everything is env-var driven, and there's exactly one code path regardless
of environment — local dev, `go test` fixtures, and production all read
the same three variables:

- `DATABASE_URL` (required, no default) — a standard
  `postgres://user:pass@host:port/dbname?sslmode=...` string. Local dev,
  CI/test, and prod differ only in which credentials/host this points at;
  see the root `README`/`docker-compose.yml` for how local Postgres gets
  started.
- `PORT` (default `8080`)
- `DIST_DIR` (default `./dist`) — where the built frontend lives; see
  below.

If you add a new required setting, add it to `config.Load()` following the
same pattern (fail fast with a clear error if a required var is missing,
`getenv(key, fallback)` for optional ones) rather than reaching for
`os.Getenv` ad hoc elsewhere in the codebase.

## Serving the frontend (`cmd/server/main.go`, `internal/api/spa.go`)

`main.go` mounts two handlers on one `http.ServeMux`: `/api/` →
`api.NewRouter(queries)`, and `/` → `api.SPAHandler(cfg.DistDir)`, which
serves static files out of `DistDir` and falls back to `index.html` for
any path that isn't a real file (so client-side navigation, if the
frontend ever adds any, doesn't 404). This is why the root `Dockerfile`
builds the frontend and backend as separate stages but ships them as one
runtime image/process — there's no separate frontend server or reverse
proxy in production.

In local dev, `vite.config.ts` proxies `/api` to `http://localhost:8080`
(see `server.proxy` in that file), so the frontend's `fetch('/api/...')`
calls work unchanged whether they're hitting the Vite dev server or the
production Go binary — same relative path either way.

## Local development

```bash
# start Postgres (see root README for docker-compose vs. native Homebrew options)
export DATABASE_URL="postgres://$(whoami)@localhost:5432/mindmap?sslmode=disable"
cd backend
go run ./cmd/server
```

The `mindmap` schema and tables are created automatically on startup
(`Migrate()`) — no separate setup step. There's no seed script yet;
`src/data/dummyData.ts` (frontend) has a realistic sample dataset that
could be adapted into one if empty-database local dev gets tedious via
the Manage UI alone.

## Deploying to production

The root `Dockerfile` builds frontend + backend into one image that serves
both from a single process — there's nothing to run separately in
production.

```bash
docker build -t mindmap .

docker run -d \
  --name mindmap \
  --restart unless-stopped \
  -p 8080:8080 \
  -e DATABASE_URL="postgres://<user>:<pass>@<db-host>:5432/<dbname>?sslmode=require" \
  mindmap
```

`DATABASE_URL` is the only thing that changes between environments — same
variable as local dev, just pointed at the real database. A few things
that only bite in production, not local dev:

- **`sslmode`** — use `require` (or `verify-full` with a CA cert) instead
  of local dev's `disable`; this connection usually crosses a real
  network, unlike the local loopback one.
- **Container-to-Postgres networking, if Postgres runs on the same
  machine as the container**: `localhost` inside the container means *the
  container*, not the host. Either run with `--network host` (Linux) and
  use `localhost`, or use the host's bridge IP / `host.docker.internal`.
  If Postgres is on a genuinely separate server (the common case), this
  doesn't apply — just use its real hostname.
- **The database and login role must already exist.** `Migrate()` (see
  above) only creates the `mindmap` *schema and tables* inside whatever
  database the connection string points at — it doesn't create the
  database or the Postgres role itself.
- **The app's DB user needs `CREATE` on the `mindmap` schema**, since
  `Migrate()` runs `CREATE SCHEMA`/`CREATE TABLE ... IF NOT EXISTS` on
  every startup.

### Setting up the app's Postgres role (prod)

Run as an admin/superuser (e.g. `postgres`), against the target database:

```sql
CREATE ROLE mindmap WITH LOGIN PASSWORD '<generate-a-strong-secret>';
CREATE SCHEMA IF NOT EXISTS mindmap;
GRANT CREATE, USAGE ON SCHEMA mindmap TO mindmap;
```

No `SUPERUSER`/`CREATEDB`/`CREATEROLE` on the role itself, and `CREATE` is
scoped to just the `mindmap` schema, not the whole database. With that in
place, `Migrate()` (running as `mindmap`) creates the tables itself on
first startup and — since it owns whatever it creates — already has full
DML on them, no separate `GRANT SELECT, INSERT, UPDATE, DELETE` step
needed. Same goes for any future table `schema.sql` adds: no default
privileges to maintain, since the app's own role is what creates it.

If Postgres itself is being stood up fresh rather than pointed at an
existing instance, `docker-compose.yml` shows the `app`+`db` shape as a
starting point — swap in real credentials and a real (non-anonymous)
volume for the data directory.

## Testing

There are no backend tests yet. If/when they're added, the natural seam is
`internal/api` handlers tested against a real local Postgres pointed at by
`DATABASE_URL` (matching how this was manually verified during
development) rather than mocking `*db.Queries` — the schema/query layer is
thin enough that a mock would mostly just re-assert the SQL, not catch
real bugs.
